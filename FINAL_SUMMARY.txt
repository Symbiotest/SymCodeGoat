================================================================================
                                THE PROBLEM
================================================================================

The authenticateUser() method was vulnerable to SQL injection attacks because
it was building SQL queries by directly concatenating user-supplied input
(username and password) into the SQL string using String.format():

    String query = String.format(
        "SELECT * FROM users WHERE username='%s' AND password='%s'", 
        username, password
    );

This allowed attackers to inject malicious SQL code through the username or
password parameters, potentially:
- Bypassing authentication (e.g., ' OR '1'='1)
- Extracting sensitive data (e.g., UNION-based injection)
- Modifying or deleting database records
- Executing arbitrary SQL commands

================================================================================
                                THE SOLUTION
================================================================================

Replaced the vulnerable string concatenation with a secure PreparedStatement
using parameterized queries:

    String query = "SELECT * FROM users WHERE username=? AND password=?";
    
    try (Connection conn = DriverManager.getConnection(DB_URL, DB_USER, DB_PASSWORD);
         PreparedStatement stmt = conn.prepareStatement(query)) {
        
        stmt.setString(1, username);
        stmt.setString(2, password);
        
        try (ResultSet rs = stmt.executeQuery()) {
            boolean authenticated = rs.next();
            logAccess(username, "login_attempt", authenticated ? "success" : "failed");
            return authenticated;
        }
    }

================================================================================
                              WHY THIS WORKS
================================================================================

1. PARAMETERIZED QUERIES
   - SQL structure is fixed and defined separately from data
   - User input cannot modify the SQL command structure

2. PARAMETER BINDING
   - User input is bound as data parameters using setString()
   - Parameters are treated as literal values, not SQL code

3. AUTOMATIC ESCAPING
   - JDBC driver automatically handles proper escaping
   - Special characters are safely escaped

4. TYPE SAFETY
   - setString() ensures parameters are treated as strings
   - No type confusion or injection possible

5. SEPARATION OF CONCERNS
   - SQL logic is completely separated from user input
   - Clear distinction between code and data

================================================================================
                            ATTACK VECTORS PREVENTED
================================================================================

✅ OR-based Authentication Bypass
   Attack: username = "' OR '1'='1"
   Result: PREVENTED - Input treated as literal string

✅ UNION-based Data Extraction
   Attack: username = "' UNION SELECT * FROM admin_users --"
   Result: PREVENTED - UNION keyword is part of parameter value

✅ Comment-based Bypass
   Attack: username = "admin' --"
   Result: PREVENTED - Both username AND password must match

✅ Stacked Queries
   Attack: username = "'; DROP TABLE users; --"
   Result: PREVENTED - Input treated as literal string value

✅ Time-based Blind SQL Injection
   Attack: username = "' AND SLEEP(5) --"
   Result: PREVENTED - Parameterized queries prevent all SQL injection
